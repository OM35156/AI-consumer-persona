## 1. 開発の全体的な流れ

```
① イシューを立てる          → GitHub の Issues で「何をするか」を宣言する
② ブランチを切る            → main から作業用ブランチを作る
③ コードを書いてコミット     → 小さな単位でこまめに保存する
④ プッシュして PR を作る    → GitHub 上でレビューを依頼する
⑤ レビュー → 修正 → 承認   → チームメンバーが確認・承認する
⑥ main にマージ             → 本線に取り込む
⑦ ブランチを削除            → 作業ブランチをクリーンアップする
```

**重要なルール:**
- `main` ブランチは常に動く状態を保つ。直接プッシュ禁止
- 必ずイシューを立ててから作業を始める
- 必ず PR を通じて main にマージする（直接マージ禁止）

## 3. ブランチのルール

### ブランチとは？

ブランチは「作業の分岐」です。本線（`main`）を壊さずに安全に開発するために使います。
作業が終わったら PR を通じて `main` に取り込みます。

```
main（本線・常に動く状態）
├── feature/42-add-rfp-parser    ← 新機能の開発
├── fix/55-fix-encoding-error    ← バグ修正
└── chore/10-update-deps         ← ライブラリ更新など
```

### ブランチの作り方

```bash
# まず main を最新にする
git checkout main
git pull origin main

# 新しいブランチを作って移動する
git checkout -b feature/42-add-rfp-parser
```

### ブランチ名のルール

形式: `<タイプ>/イシュー番号-作業内容の短い説明`

| タイプ | 用途 | 例 |
|---|---|---|
| `feature/` | 新機能追加 | `feature/42-add-rfp-parser` |
| `fix/` | バグ修正 | `fix/55-fix-encoding-error` |
| `chore/` | 依存更新・設定変更・リファクタリング | `chore/10-update-deps` |

> イシュー番号を含めることで、どの作業のブランチか一目でわかります。

## 4. コミットのルール

### コミットとは？

コミットは「変更内容の保存」です。適切なメッセージを付けることで、
後からいつ・何を・なぜ変更したかが追跡できます。

### コミットメッセージの形式

```
<タイプ>: <何をしたかを日本語で説明>
```

| タイプ | 用途 | 例 |
|---|---|---|
| `feat` | 新機能追加 | `feat: RFPパーサーに複数ファイル対応を追加` |
| `fix` | バグ修正 | `fix: CSVパーサーのエンコーディングエラーを修正` |
| `refactor` | 動作を変えないコード整理 | `refactor: データ取得ロジックをクラスに分離` |
| `test` | テスト追加・修正 | `test: APIクライアントのユニットテストを追加` |
| `docs` | ドキュメント変更 | `docs: READMEにセットアップ手順を追記` |
| `chore` | 依存更新・設定変更 | `chore: 依存パッケージを更新` |

### コミット前のチェック

```bash
# テストが通るか確認
uv run pytest

# コードスタイルチェック
uv run ruff check .
```

> 1コミット = 1つの論理的な変更にまとめる。「あれもこれも」を1コミットに詰め込まない。

## 5. イシュー（Issue）の使い方

### イシューとは？

イシューは「やること・問題・要望」を記録する場所です。
作業を始める前に必ずイシューを立てて、チームに「これをやります」と宣言しましょう。

### イシューの立て方

1. GitHub の **Issues** タブ → **New issue** をクリック
2. テンプレートを選ぶ（バグ報告 or 機能要望）
3. 必要事項を記入して **Submit new issue**
4. 作成されたイシューに**自分をアサイン**する（右側の Assignees）

### イシューとブランチの紐づけ

イシュー番号をブランチ名に含めることで自動的に紐づきます。

```bash
# イシュー #42 の作業をする場合
git checkout -b feature/42-add-rfp-parser
```

## 6. PR（プルリクエスト）の手順

### PR とは？

PR（プルリクエスト）は「この変更を main に取り込んでください」というレビュー依頼です。
他のメンバーがコードを確認・承認してから初めてマージできます。

### 手順

#### ① 変更をプッシュする

```bash
# 変更をステージング
git add <変更したファイル>

# コミット
git commit -m "feat: 〇〇機能を追加"

# GitHub にプッシュ
git push origin feature/42-add-rfp-parser
```

#### ② GitHub で PR を作成する

1. GitHub のリポジトリページを開くと「Compare & pull request」ボタンが表示される
2. クリックして PR 作成画面を開く
3. **テンプレートの全項目を埋める**
4. `Closes #42` のように関連イシュー番号を記載する（マージ時に自動クローズされる）
5. **Reviewers** に最低1名を指定する
6. まだ作業中なら **Draft pull request** として作成する

#### ③ レビュー対応

- レビューコメントへの返信・修正を行う
- 修正したら追加コミット → プッシュ（PR は自動で更新される）
- 全ての指摘に対応したら **Re-request review** を押す

#### ④ マージ

- レビュアーが承認（Approve）したらマージ可能
- マージ方式は **Squash and merge** を選ぶ（複数コミットを1つにまとめてマージ）
- マージ後は **Delete branch** でブランチを削除する

### PR 作成前のセルフチェックリスト

- [ ] 自分で差分（diff）を確認した
- [ ] `uv run pytest` が通る
- [ ] APIキーや機密情報がコードに含まれていない
- [ ] PR テンプレートの全項目を埋めた

## 7. コードレビューの観点

レビュアーは以下の観点で確認してください。

| 観点 | チェック内容 |
|---|---|
| 正確性 | 要件を満たしているか、バグがないか |
| セキュリティ | APIキーのハードコード禁止、SQLインジェクション等がないか |
| コードスタイル | 型ヒント・日本語コメント・PEP 8 準拠 |
| テスト | 主要なロジックにテストがあるか |

**レビューする側へ:** 指摘は「問題点＋改善案」をセットで伝えましょう。

**レビューされる側へ:** コードへの指摘はコードへの批判ではありません。
疑問点は素直に聞いて、チームで解決しましょう。

## 8. セキュリティルール

- **APIキー・パスワード・トークンをコードに直接書かない**
- 機密情報は `.env` ファイルで管理する（`.gitignore` に必ず含める）
- コミット前に機密情報が含まれていないか確認する
- セキュリティの脆弱性を発見した場合は、公開イシューではなく**直接担当者に連絡**する

> `.env` ファイルを誤ってコミットしてしまった場合は、すぐに担当者に報告してください。
> git の履歴から完全に削除する対応が必要です。
